# 03. Arrays (配列)

## 制約 (Constraints)
*   **データ構造の配列化 (カラムナ/列指向):**
    オブジェクト（`{ id: 'PRD-01', name: 'ノートPC', ... }`）をカプセル化した配列として持つのではなく、属性ごとの独立したフラットな配列（`productIds`, `productPrices` など）に分解して保持します。
*   **ループの排除と宣言的記述:**
    要素ごとに逐次処理を行う命令的なループ構造（for, whileなど）の利用を極力制限します。代わりに、配列の高階関数（`map`, `filter`, `reduce`, `forEach`）を使用して、配列全体に対する変換と縮約の形でロジックを表現します。

---

## コードの特徴・解説
今回の実装では、データを以下のように列（カラム）単位の独立した配列として定義しました。

```typescript
const productIds = ['PRD-01', 'PRD-02', 'PRD-03'];
const productNames = ['ノートPC', 'マウス', 'キーボード'];
const productPrices = [100000, 3000, 5000];
const productStocks = [2, 5, 0];
```
異なるデータ配列同士は「配列内のインデックス」によって関係性が保持されます。例えば、商品IDの `productIds.indexOf('PRD-01')` が `0` であれば、その名前は `productNames[0]`、価格は `productPrices[0]` になります。

チェックアウト処理（`checkout`）では、命令的なループを一切使用せず、配列関数を組み合わせて実装しています：
1.  **`map`:** カートのID配列をベースに、小計や名前をまとめたレシート行オブジェクト配列へ一括変換。
2.  **`forEach`:** レシート行を出力。
3.  **`reduce`:** 全アイテムの小計を足し合わせ、合計金額に縮約（Reduction）。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**データを「個々のオブジェクト」ではなく「データの集まり（ベクトル・行列）」として捉え、一括処理すること**です。
関係データベースのテーブルを縦方向に切り出した「カラムナ（列指向）データベース」や、ビッグデータ解析で用いられるデータフレーム（Pandasなど）のモデルに非常に近いです。オブジェクト指向のように「個」を意識するのではなく、「集合」に対して演算を適用するという思考モデルの転換が求められます。

### 難しかった点 / 気づき
*   **インデックス同期の脆弱性:**
    データが物理的に複数の配列に分かれているため、1つの配列の順序が狂ったり、要素の追加・削除時に同期を誤ったりすると、データ全体の整合性が簡単に崩壊します（例: ID配列の要素を削除したのに価格配列を削除し忘れる、など）。このため、実装上の注意深さが必要になります。
*   **コードの宣言性と美しさ:**
    `map` や `reduce` を用いることで、一時的なループカウンタや変数を用意する必要がなくなり、データの流れ（パイプライン）が非常に綺麗に表現できます。一時変数に状態を溜め込まないため、バグが混入しにくいという強みを感じました。

### 実務への応用
*   **カラムナデータベース (BigQuery / Redshift):**
    行指向ではなく列指向でデータを格納することで、大量データの集計クエリを劇的に高速化する仕組みの基礎概念を学べます。
*   **ビッグデータ処理 / MapReduce:**
    配列を `map` して `reduce` するパターンは、大量データを複数マシンに分散して処理する「MapReduce」アーキテクチャのモデルそのものです。
*   **グラフィックス・数値計算 (GPGPU):**
    Shaderプログラムや3Dグラフィックス、機械学習など、SIMD（シングルインストラクション・マルチデータ）演算が要求される分野での配列指向プログラミングに直結します。
