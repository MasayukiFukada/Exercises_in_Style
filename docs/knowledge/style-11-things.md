# 11. Things (シングズ / モノ - オブジェクト指向)

## 制約 (Constraints)
*   **モノのオブジェクト化:**
    現実世界の「モノ（Things）」をクラスおよびオブジェクト（インスタンス）として表します。
*   **カプセル化 (Data Encapsulation):**
    データ（状態）と、そのデータに対する手続きを同じオブジェクト内にまとめます。
*   **ゲッターとセッターを通じた操作:**
    オブジェクトのプロパティ（内部状態）に直接外部からアクセスしたり改変したりするのを避け、公開されたメソッド（ゲッター、セッター、振る舞い用メソッド）を介して状態の問い合わせや変更を行います。

---

## コードの特徴・解説
今回の実装では、TypeScriptの `class` 構文を用いて、ECショッピングカートのドメインを表現するオブジェクトを作成しています。

*   **カプセル化されたクラス定義:**
    `Product` クラスは `id` や `stock` などのプロパティを `private` に隠蔽し、在庫の変更は `decreaseStock()` メソッドを通じてのみ安全に行えるようにしています。
    ```typescript
    class Product {
      private readonly id: string;
      private stock: number;
      // ...
      decreaseStock(qty: number): void {
        if (!this.hasStock(qty)) {
          throw new Error("在庫不足");
        }
        this.stock -= qty;
      }
    }
    ```
*   **関心の分離:**
    カタログ (`Catalog`)、ショッピングカート (`ShoppingCart`)、および決済システム (`CheckoutService`) がそれぞれ独立した「モノ」として振る舞い、協調動作します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**データと振る舞いを同一箇所に束ねる（Cohesion / 凝集度を高める）こと**です。
モジュールや関数にデータを切り離して渡す手続き型とは異なり、オブジェクト自身が自分の状態の妥当性（例: 在庫数があるか）を検証し、自身の状態を更新します。これにより、ビジネスルールが散散にならず、変更に強い構造になります。

### 難しかった点 / 気づき
*   **状態の変更管理:**
    これまで実装してきた `Pipeline` や `TheOne` (モナド) などの関数型スタイルでは、イミュータブル（不変）な状態の受け渡しを行っていましたが、本スタイルではオブジェクトが内部に可変（Mutable）な状態を保持し、メソッド呼び出しによって自己の状態を書き換えます。どちらの手法にもメリット・デメリットがあり、オブジェクト指向では直感的ですが、複数の場所から同じオブジェクトが共有されて変更される際の副作用（状態の予測しにくさ）に気をつける必要があります。

### 実務への応用
*   **ドメイン駆動設計 (DDD):**
    ドメインモデル（エンティティや値オブジェクト）を定義する際の基礎となります。データだけの構造体（ドメインアネミックモデル）を避けて、適切なビジネスロジックをクラスにカプセル化することが、保守性の高い設計に繋がります。
