# 40. Recurrent (リカレント - RNN)

## 制約 (Constraints)
- **状態の直接保持禁止**: カートの中身や合計金額などの「状態」を、直接書き換え可能なメンバ変数やグローバル変数として保持しない。
- **入力としての時系列シーケンス**: カートへの追加操作などのイベント（操作、商品インデックス、要求数量など）を要素とする、時系列のシーケンスデータとして操作履歴を管理する。
- **リカレントセル (RNN Cell) の定義**: 前回の隠れ状態（$h_{t-1}$）と現在の入力（$x_t$）を受け取って新しい隠れ状態（$h_t$）を出力する純粋な遷移関数を定義する。
- **最終隠れ状態からの決定**: システムの意思決定（在庫十分性のチェックや割引の適用、レシート出力）は、操作シーケンス全体を RNN に流し込んだ結果得られる「最終ステップの隠れ状態 $h_T$」に基づいてのみ行う。

## コードの特徴・解説
- `InputVector` を `[is_add_action, product_index, request_qty]` として定義し、カートへの追加操作をすべて時系列のイベントシーケンス `actionHistory` として保持しています。
- `rnnCell` 関数は、前回の隠れ状態 `hPrev` と新しい入力 `x` を受け取ります。隠れ状態の中身は `[qty_prd0, qty_prd1, qty_prd2, error_flag, error_prd_idx, accum_subtotal]` であり、各商品のカート内数量やエラーフラグ、累積金額を保持しています。
- カートへの追加試行時には、一時的な候補履歴（`candidateHistory`）を作成して `runRnn` に通し、最終的な隠れ状態の `error_flag` が `0` (判定OK) なら正式に履歴として採用し、`1` (判定NG) なら却下してエラーメッセージを出力します。これにより、状態の変更は「履歴シーケンスへの要素追加」のみによって表現されます。
- チェックアウト時の割引適用判定も、最終隠れ状態 `hFinal[5]` に蓄積された累計小計に対して、割引判定ネットワークを適用して計算します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
*   **RNNのデータフロー:** 前回の隠れ状態と現在の入力を受け取って新しい状態を出力する純粋な遷移関数（RNNセル）のシーケンス実行を模しています。
*   **イベントソーシングとの酷似:** 現在の状態を直接保存するのではなく、発生した全操作イベント（履歴）を最初から再生（リプレイ）することで最新の状態を復元する設計は、現代的なシステムアーキテクチャである「イベントソーシング」と同一の哲学です。

### 難しかった点 / 気づき
*   **エラー時の状態ハンドリング:** 在庫切れなどの更新エラーを検知して状態の更新を拒否する挙動を、RNNセル内の状態遷移と「候補シーケンスの評価」というモデルに落とし込む設計がパズル的でした。
*   **実行状態の数式化:** システム全体のすべての変数の変化が、線形結合と活性化（ステップ関数）の繰り返しによる数学的なシーケンス計算に完全に定式化される点が新鮮でした。

### 実務への応用
*   **時系列パターン検出と分析:** 金融取引履歴からの不正検出、ユーザーの購買遷移ステップの追跡、対戦ゲームにおけるコマンド入力シーケンスのAI判定など、時間経過や順序に強く依存するロジックの実装に有効です。
*   **状態管理フレームワーク:** Reduxにおける `Action` と `Reducer` の関係や、分散システムにおけるイベントストリームの処理ロジックの理解に直接役立ちます。
