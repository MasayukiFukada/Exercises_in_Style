# 01. Good Old Times (古き良き時代)

## 制約 (Constraints)
*   **極端に制限されたメモリ:**
    状態の保持や処理に必要な領域は、1つの巨大なグローバル配列（物理メモリの模擬、サイズ 1024 など）のみに制限されます。
*   **名前付き変数の禁止 (グローバルデータスペースの直接参照):**
    コード中で名前付き変数を状態保持やポインタとして宣言することは禁止されています。すべてのデータ読み書きは `data[200]` のような「メモリのアドレス番号（インデックス）」を指定して直接行います。
*   **関数引数やローカル変数の排除:**
    サブルーチン（関数）の呼び出しは可能ですが、パラメータの受け渡しや戻り値の返却は関数の引数ではなく、決められたメモリ領域（レジスタに見立てたインデックス）を経由して行わなければなりません。

---

## コードの特徴・解説
今回のECショッピングカートの実装におけるメモリレイアウト設計は以下の通りです：

*   **カタログ領域 `[0 - 99]`:** 最大3商品。各商品 `[商品ID, 商品名, 単価, 在庫数]` の4スロットずつ使用。
*   **カート領域 `[100 - 199]`:** `data[100]` にカート内商品数を保持。`data[101]` 以降に `[商品ID, カート内数量]` を格納。
*   **一時変数・レジスタ領域 `[200 - 299]`:** ポインタや中間計算結果を保持。
    *   `data[200]`, `data[209]`: ループカウンタ
    *   `data[202]`, `data[203]`: サブルーチンの引数（商品ID、追加数量）
    *   `data[205]`: エラーバッファ（エラーメッセージ）
    *   `data[207]`, `data[208]`: 合計額、割引額などの中間計算用

サブルーチン `add_to_cart_routine` は、`data[202]` と `data[203]` からパラメータを読み取り、条件分岐とループを駆使して在庫チェックとカート情報を更新します。処理が失敗した場合は `data[205]` にエラー文字列を書き込み、呼び出し元がそれを検知してコンソールに出力します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**アセンブラや初期のFortranのような非構造化プログラミングをエミュレートすること**にあります。
現代のプログラミングでは、変数名やクラス、スコープによるデータの隠蔽（カプセル化）が当たり前ですが、それらはすべて「人間にとっての抽象化」です。メモリ空間という平坦な世界（Flat Memory Space）において、CPUやレジスタがどのようにデータをやり取りしているかを再認識させてくれます。

### 難しかった点 / 気づき
変数名が使えないため、**「どのメモリ番地に何のデータを格納したか」を正確に頭の中で管理（あるいは設計シートにマッピング）しなければならない点**が非常に困難でした。
例えば、ループの中で一時的に値を上書きしてしまうバグ（変数の競合）が簡単に発生します。`data[200]` をインデックスとして使っている最中に、別の処理で `data[200]` を書き換えてしまうとループが崩壊します。現代のコンパイラやインタープリタがスコープ（Scope）によっていかにプログラマをこのミスから保護しているかが痛感できました。

### 実務への応用
*   **低レイヤ開発・組み込みシステムへの理解:** Web開発など高度な抽象化レイヤにいると意識しにくい、メモリの断片化やメモリアドレス操作（ポインタ）に対する理解が深まります。
*   **Wasm（WebAssembly）などの線形メモリの設計:** WebAssemblyでメモリバッファ（Linear Memory）を直接JavaScript側と共有して高速にデータ転送する際、このスタイルのような「オフセットを用いたメモリアレイ設計」が実務でも役立ちます。
