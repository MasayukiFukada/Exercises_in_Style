# クラスとオブジェクト指向のパラダイムシフト：これからのプログラミングの文体

## 1. はじめに：クラスが覇権を握った歴史的背景
過去数十年間、Java、C++、C#といった「クラスベースオブジェクト指向」は、ソフトウェア開発における支配的な文体（スタイル）であり続けました。クラスがこれほどまでに普及した理由は、主に以下の3点に集約されます。

1.  **カプセル化（状態と振る舞いの結合）:** 
    無秩序なグローバル変数の書き換えを排し、関連するデータ（状態）とそれを操作するメソッド（振る舞い）をオブジェクトという最小単位に閉じ込め、不整合を防ぐため。
2.  **多態性（ポリモーフィズム）の標準化:**
    インターフェースや継承を用いることで、呼び出し側のコードを変更せずに、実行時の具体的な挙動を綺麗に差し替える設計パターンを定着させたため。
3.  **現実世界のメンタルモデルの投影:**
    「商品」「カート」「顧客」といったドメイン概念をそのままコード上の「オブジェクト」としてマッピングできるため、直感的に設計しやすかった。

このプロジェクトにおける [11. Things (オブジェクト指向)](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-11-things.md) や [14. Abstract Things (依存性注入)](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-14-abstract-things.md) は、この伝統的かつ成熟した文体を用いてECショッピングカートを整理する強力さを証明しています。

## 2. クラスベースオブジェクト指向の「影」と限界
しかし、システムの複雑化や並行処理の必要性が高まるにつれ、クラスという文体のデメリット（影）が浮き彫りになってきました。

*   **共有された可変状態 (Shared Mutable State) の罠:**
    クラスは「状態を内部に隠蔽し、隠れて書き換える」ことを基本とします。しかし、非同期処理やマルチスレッド環境において、複数のスレッドが別々の経路から同じクラスインスタンスを書き換えることで、予測不能な「競合状態（Race Condition）」を引き起こし、バグの最大の温床となりました。
*   **「バナナを頼んだら、ジャングルとゴリラが付いてきた」問題:**
    継承（Inheritance）によるコード再利用は、親クラスと子クラスの間に強力な密結合を生み出します。一部の機能を使いたいだけなのに、基底クラスの巨大な依存関係や不要なプロパティをすべて引き受けなければならず、コードの肥大化と硬直化を招きました。
*   **テストの難易度上昇:**
    クラスインスタンスが「内部のプライベート状態」を累積させていくため、単体テストを実行する際に、事前にそのオブジェクトを特定の複雑な中間状態にセットアップしなければならず、モックの乱用とテストコードの複雑化を引き起こしました。

## 3. 「クラスレス」な文体への回帰（本プロジェクトでのアプローチ）
本プロジェクトで実装した独自拡張を含む全46のスタイルでは、クラスという手段を用いずに「状態遷移」や「ポリモーフィズム」をクリーンに表現する手法をいくつも実践しました。

*   **データと振る舞いの完全分離 ([06. Pipeline](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-06-pipeline.md) / [42. Point-free](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-42-point-free.md)):**
    データを「不変（Immutable）な単なるレコード」とし、ロジックを「純粋関数（Pure Functions）」とします。状態を隠すのではなく、「古い状態を受け取り、新しい状態を返す」関数の連鎖として表現することで、副作用や並行処理の競合を根本から排除します。
*   **クロージャによる隠蔽 ([13. Closed Maps](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-13-closed-maps.md)):**
    クラス定義を排除し、関数のレキシカルスコープ（クロージャ）を用いてアドホックに状態を閉じ込めます。これにより、重厚なクラス構文を持たない軽量な言語でも、安全な隠蔽を実現します。
*   **イベントの蓄積と投影 ([43. Event Sourcing](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-43-event-sourcing.md) / [46. Reactive Streams](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-46-reactive-streams.md)):**
    状態の「更新」そのものを禁止します。状態の変更は「過去の不変のイベント」としてストリームに追記されるだけであり、現在の状態はそれらのリプレイ（scan）によってのみ動的に再現されます。これにより、状態の破壊的更新という概念自体が消滅します。

## 4. 現代の実務開発におけるクラスの退潮と「関数型」の復権
実務のシステム開発においても、クラスへの依存度は年々下がっており、関数型・イミュータブルな設計思想が主流を占めつつあります。

1.  **フロントエンドにおける「関数＋フック」への移行:**
    Reactはかつてクラスコンポーネントが標準でしたが、現在は「関数コンポーネントと不変ステート（Hooks）」に完全移行しました。これにより、`this`の参照問題が解消され、ロジックの再利用性が劇的に向上しました。
2.  **モダン言語（Go / Rust）における継承の排除:**
    GoやRustには `class` キーワードが存在しません。データ（構造体 `struct`）とメソッドは分離して定義され、ポリモーフィズムは継承ではなく「インターフェース」や「トレイト」といった共通の契約定義によってのみ実現されます。
3.  **サーバーレス・マイクロサービスとの相性:**
    クラウドネイティブなインフラでは、実行インスタンスがミリ秒単位で起動・消滅するため、メモリ上に状態を抱え込むクラスベースの設計（Stateful）は嫌われます。状態を持たない（Stateless）処理を純粋関数で行うことこそが、スケーラビリティの鍵となっています。

## 5. 生成AI時代における「文体」のパラダイムシフト
さらに、生成AI（LLM）にコードを生成させ、AIエージェントと自律開発を行う現代において、クラスを避ける「クラスレス・関数型スタイル」は決定的な優位性を持っています。

*   **LLMにとっての「状態隠蔽」の難しさ:**
    クラスオブジェクトが内部でどのようなプライベート状態を持っているかは、静的なコード解析だけでは追跡が困難です。AIも人間と同様に、暗黙的な状態の変化（副作用）を予測するのが苦手であり、バグを混入させやすくなります。
*   **「明示的な型」と「純粋関数」の圧倒的優位性:**
    「入力と出力の型が厳密に定義され、副作用を持たない関数（Pipeline）」は、コンテキストがその場（局所的）で完結しているため、LLMが最も正確にコードを生成でき、AIエージェントによる自動テスト・デバッグのループも容易に回りやすくなります。
*   **名前（仮引数や変数）の重要性:**
    AIはコードの「名前（意味）」を頼りに推論を行います。[42. Point-free](file:///home/minamo/repository/exercises_in_style/docs/knowledge/style-42-point-free.md) のように名前を一切排除する極端なスタイルはAIを混乱させますが、適切な型と純粋関数で組まれたクラスレス設計は、AIにとっても人間にとっても最高の「読み書きの文体」となります。

---

## 6. 結び：未来のアーキテクトに求められる「文体の選択」
以上の考察から、**「クラスは現代のプログラミングにおいて、もはや唯一の正解ではなく、数ある設計の選択肢（一文体）に過ぎない」**ということが言えます。

今後、プログラミング言語やインフラ、そしてAIとの協調が進化するにつれて、アーキテクトには以下のような柔軟な「文体の選択」が求められます。

*   **クラスを選択すべき場面:** GUIコンポーネント、ゲームエンジン、物理シミュレーションなど、「独自の可変状態を持った無数のオブジェクトが相互作用する」ドメイン。
*   **クラスを避けるべき（関数型・ストリーム型を選択すべき）場面:** Web API、データ処理パイプライン、分散マイクロサービスなど、「入力データを加工して安全に転送・保存する」ドメインや、高い並行性が求められるシステム。

『Exercises in Programming Style』という文体練習の実践は、クラスという慣れ親しんだ当たり前の道具を一時的に奪う（あるいは極端に適用する）ことで、私たちに**「パラダイムの本質を見極める目」**を与えてくれます。これからの開発においては、ドメインの本質や開発体制（AIとのペアプログラミングを含む）に応じて、最もバグが少なく美しい「文体」を能動的に選択していくことが極めて重要です。
