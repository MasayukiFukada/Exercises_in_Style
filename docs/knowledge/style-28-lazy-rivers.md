# 28. Lazy Rivers (ゆったり流れる川 / 遅延評価)

## 制約 (Constraints)
*   **遅延評価の徹底 (Lazy Evaluation):**
    データを配列などに一度にすべて展開して一括処理することを禁止します。必要な時に必要なデータだけを取り出して処理します。
*   **ジェネレータ / イテレータの利用:**
    処理の各ステップをジェネレータ関数（`function*` と `yield` キーワード）として記述し、イテラブルな「データの川（ストリーム）」を繋ぎます。
*   **末端評価による駆動:**
    下流の最終消費者（`for-of` や `next()`）がデータを要求するまで、上流の処理（カートへの追加やバリデーション）は一切進行しないように設計します。

---

## コードの特徴・解説
今回の実装では、カタログの印刷、カートへの追加、チェックアウトレシートの生成と在庫の減算にいたるまで、すべてジェネレータ関数で構成しています。

*   **遅延カートローダー:**
    追加リクエストを1件ずつ処理し、有効なアイテムのみを yield します。
    ```typescript
    function* cartLoader(requests: Iterable<AddRequest>, products: Product[]): Generator<CartItem> {
      for (const req of requests) {
        // ... 在庫検証 ...
        yield { product: p, qty: req.qty };
      }
    }
    ```
*   **パイプラインの結合:**
    リクエストを受け取ってカートを構築するストリーム（`cartStream`）と、それを消費してレシートを生成するストリーム（`receiptPipeline`）を結合しますが、この時点では一切計算は行われません。
*   **末端での評価開始:**
    最後の `for (const outputLine of receiptPipeline)` が動いて初めて、カート追加のログやレシート明細の行が1ステップずつ同期的に処理され、コンソールへ流れ出します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**メモリ消費の最小化と、データ処理の無限ストリームへの対応**です。
もし何万点もの購入リクエストや、巨大なログデータを一括で配列にロードして処理しようとすると、メモリが枯渇してクラッシュします。遅延評価は、データを「ゆったり流れる川」のように1件ずつ小分けにしてパイプラインを流すため、メモリ消費を常に一定に抑えることができます。

### 難しかった点 / 気づき
*   **副作用の発生タイミングの制御:**
    「在庫を減算する」という不純な副作用が、イテレータのループが回りきったタイミングで初めて起きるように設計する必要があります。ループを途中でブレイクすると在庫が引かれないといったバグを生む可能性があるため、遅延評価パイプラインの中での副作用の織り込み方（トランザクション性）には細心の注意が必要です。

### 実務への応用
*   **巨大ファイルのストリーム処理:**
    Node.js の `fs.createReadStream()` や、CSVパーサーのストリーム機能を用いて、数ギガバイトある大容量ファイルをメモリを汚さずに1行ずつパースしてDBへ保存するようなバッチ処理の実装に最適です。
*   **LINQ / RxJS / Java Streams:**
    C#の `IEnumerable` (LINQ)、Javaの `Stream` API、RxJS など、宣言的にデータの変形をパイプラインで記述し、ターミナル操作（`.collect()`, `.subscribe()` など）が呼ばれた際に初めて背後の実行エンジンが遅延動作する現代の標準的なデータ処理パターンとなっています。
