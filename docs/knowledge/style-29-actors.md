# 29. Actors (アクター)

## 制約 (Constraints)
*   **共有状態の完全排除:**
    各アクター（実体オブジェクト）は、自身の中に状態（データ）を完全に閉じ込め、他のアクターと一切メモリ上の状態を共有しません。
*   **直接的メソッド呼び出しの禁止:**
    アクター同士の同期的なメソッド呼び出し（例: `cart.addItem()`）を完全に禁止します。すべての対話は非同期メッセージ送信（メールボックスへの投稿）のみで行います。
*   **メールボックスと自律スレッド:**
    各アクターは独自の「メールボックス（キュー）」を持ち、そこに届いたメッセージを自律的に1件ずつ順番（シングルスレッド風）に処理するイベントループを内包します。

---

## コードの特徴・解説
今回の実装では、キューによる非同期処理を再現する `Actor` 基底クラスを自作し、`setTimeout`（イベントループへの登録）によって完全な非同期メッセージ駆動をシミュレートしています。

*   **アクター基底クラス `Actor`:**
    メッセージの `post()`（投稿）と、キューに溜まったメッセージを順次処理する非同期ループを実装しています。
    ```typescript
    abstract class Actor {
      private mailbox: any[] = [];
      post(message: any): void {
        this.mailbox.push(message);
        // 非同期にループをトリガー
        setTimeout(() => this.processMailbox(), 0);
      }
    }
    ```
*   **メッセージ送信による協調:**
    シナリオ進行役の `ScenarioActor`、在庫管理の `CatalogActor`、カートの `CartActor` が、互いのメソッドを直接知ることなく、メッセージのやり取り（`catalogActor.post(['check_stock', { ... }])`）だけでカート追加やチェックアウトを完了させます。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**並行処理（スレッド安全）の複雑さを、非同期メッセージングによって「カプセル化（隔離）」すること**です。
通常のスレッドプログラミングでは、複数のスレッドから同時に同じ変数にアクセスされる「競合（Data Race）」を防ぐため、複雑なロック制御（Mutexなど）が必要になります。アクターモデルでは、状態に触れることができるのはアクター自身のみ（シングルスレッド処理）で、外部とはコピーしたメッセージのみで会話するため、ロックを一切使わずに安全な並行処理を記述できます。

### 難しかった点 / 気づき
*   **制御フローの非同期・イベント化:**
    同期呼び出しと戻り値がないため、「商品を追加して、その成否を受けて次の商品を追加する」という直列ストーリーを実行するだけでも、`stock_checked_result` $\rightarrow$ `add_success` $\rightarrow$ `proceedScenario` のように、コールバックと状態遷移のバケツリレーを実装する必要があり、制御の記述難易度は上がります。

### 実務への応用
*   **Erlang / Elixir (OTP) や Akka (Scala/Java):**
    アクターモデルを言語機能やフレームワークとして標準サポートしており、数百万アクターが並行稼働するような耐障害性の極めて高い大規模通信サーバーや分散ゲームサーバーなどで中核を担っています。
*   **Web Workers (ブラウザ並行処理):**
    JavaScriptにおいて、UIをロックせずに重い計算を別スレッドで走らせる Web Workers も、`postMessage()` を介してメッセージをやり取る完全なアクターモデルの設計となっています。
