# 21. Constructivist (構造主義者 / 防御的)

## 制約 (Constraints)
*   **プログラム異常終了の禁止:**
    想定外のデータや制約違反（在庫切れ、ID未登録など）を検知しても、例外（`throw`）を投げたり、プログラムを異常終了させてはなりません。
*   **健全なフォールバック:**
    違反を検知した場合は、最も安全なデフォルト値や、その時点で受け入れ可能な最大値（例: 在庫不足時は現在ある在庫の最大数）に自動で補正して処理をそのまま実行し続けます。

---

## コードの特徴・解説
今回の実装では、在庫不足時に例外を出さず、引き当て可能な最大数を自動計算してカートへ追加し、処理を完了させています。

*   **引き当て上限の自動適用:**
    カート追加時、要求された数量より在庫が少なければ、自動的に在庫数に丸めてカートに追加します。
    ```typescript
    let addAmount = safeQty;
    if (availableStock < currentInCart + safeQty) {
      addAmount = availableStock - currentInCart;
      console.log(`[警告] 在庫不足のため、引き当て可能な ${addAmount}個 のみ追加します。`);
    }
    ```
*   **ダミーオブジェクトへのフォールバック:**
    商品情報（`Product`）が空（`undefined`）の場合でも、ダミーの代替商品を自動的に補完することで、処理のクラッシュを防いでいます。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**システムのレジリエンス（回復力）の最大化**にあります。
エラーが起きたからといって即座に処理全体を破棄するのではなく、システムとして可能な限りの最善を尽くして（Best Effort）トランザクションを完了に導くという方針です。

### 難しかった点 / 気づき
*   **不具合の見逃しリスク:**
    例外をスローしないため、内部的な不整合やプログラムのバグが「隠蔽」され、後から気づきにくくなるリスクがあります。実務でこのアプローチをとる際は、ログや警告メッセージの出力、あるいは監視用のカウンタなどを適切に組み合わせて、裏側で何が補正されたかを検知できるように設計する必要があります。

### 実務への応用
*   **フォールバック付きの割引計算やレコメンデーション:**
    ECサイトなどで、割引計算システムや商品推薦（レコメンド）APIが一時的にクラッシュした際、購入処理全体をエラーにするのではなく「割引なし」「デフォルトの人気商品一覧」にフォールバックして、ユーザーがチェックアウト（購入）を完了できるように設計します。
*   **堅牢なファイルパーサー:**
    CSVやXMLなどの読み込み時に、一部の行のフォーマットが崩れていても全体のエラーとせず、その行のみスキップしたりデフォルト値を補完して読み込みを完了させます。
