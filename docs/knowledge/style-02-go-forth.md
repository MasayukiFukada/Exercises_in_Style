# 02. Go Forth (前へ進め - スタック指向)

## 制約 (Constraints)
*   **グローバルなデータスタックの利用:**
    値の受け渡し（関数のパラメータや戻り値）は、原則としてすべてグローバルな「データスタック」を介して行われます。
*   **サブルーチン（ワード）のシグネチャ排除:**
    関数（ワード）は引数リストを持たず、戻り値を直接 `return` しません。実行前にスタックから値を `pop` し、結果がある場合はスタックに `push` します。
*   **ヒープ領域による永続的な状態保持:**
    カタログやカートのデータなど、スタック上で一時的に扱えない永続的な状態は、「ヒープ（グローバル辞書/オブジェクト）」に格納して管理します。
*   **ポストフィックス（後置記法）風のフロー制御:**
    「引数をスタックに積む $\rightarrow$ 命令を実行する」という順番で処理を記述します。

---

## コードの特徴・解説
今回の実装では、グローバルに `stack`（配列）と `heap`（オブジェクト）を用意しました。

*   **引数の渡し方:**
    例えばカート追加の際、ID `'PRD-02'` と数量 `1` をスタックに積んでから、ワード `add_to_cart_word` を呼び出します。
    ```typescript
    push('PRD-02');
    push(1);
    add_to_cart_word();
    ```
*   **サブルーチン内部の動作:**
    `add_to_cart_word` の中では、スタックの性質（LIFO: 後入れ先出し）に従い、最後に積まれたものから順にポップします。
    1.  `qty = pop()` （数量をポップ、上にあるため最初に取り出せる）
    2.  `id = pop()`  （商品IDをポップ）
    処理結果として、処理の成功可否を表す真偽値（フラグ）をスタックに `push(true)` または `push(false)` し、エラーがあれば `heap['error']` にメッセージを設定します。
*   **検証と出力:**
    結果検証ワード `verify_result_word` は、スタックから成否フラグを `pop()` して検証し、失敗していればヒープからエラーを取り出してコンソールに出力します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**関数の名前（ワード）とデータの受け渡しを極限まで疎結合にすること**です。
現代の言語では、関数を呼ぶ際に `add_to_cart(id, qty)` のように引数の数や型が厳しくチェックされますが、スタック指向では関数は「ただスタックの上にあるデータを処理する装置」に過ぎません。この柔軟さが極限のシンプルさをもたらしています。

### 難しかった点 / 気づき
*   **スタックの順番（LIFO）の管理:**
    引数を渡す順番と、関数内で取り出す（ポップする）順番が逆になるため、常にスタックの「現在の状態（何がどこに積まれているか）」をプログラマが頭の中で把握していなければなりません。引数の順番を1つ間違えるだけで、全く異なるデータが関数に渡り、バグの原因になります。
*   **関数の呼び出し順の重要性:**
    スタック指向では「状態」がスタック自体に蓄積されていくため、関数の呼び出し順序がそのままデータのフローになります。現代のプログラムに比べて「何が何に依存しているか」が可視化されにくいスリリングさがありました。

### 実務への応用
*   **仮想マシンの設計:**
    JavaのJVM（Java Virtual Machine）や、Pythonのバイトコードインタープリタ、PostScript、WebAssemblyなど、多くのランタイムは「スタックマシン」として動作しています。このスタイルを理解することは、これらのランタイムが低レベルでどのように命令を評価しているかを理解する強固な基盤になります。
*   **逆ポーランド記法 (RPN) による計算機の構築:**
    電卓アプリケーションや数式パーサーを開発する際、スタックを用いた字句解析・数式評価ロジックが直感的に設計できるようになります。
