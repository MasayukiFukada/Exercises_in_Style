# 43. Event Sourcing (イベントソーシング)

## 制約 (Constraints)
- システムの現在の状態（商品の在庫やカートの中身など）を直接更新する再代入や破壊的操作を禁止する。
- 状態の変更はすべて「過去に起きた不変の事実」を表す「イベント」オブジェクトとして定義され、イベントログ（Append-Only）に記録される。
- 現在のアプリケーション状態を取得したい場合は、初期状態に対してイベントログの全履歴を順に適用（Replay / 畳み込み）して動的に構築する。

## コードの特徴・解説
- アプリケーション全体の状態管理は `eventStore` 配列（イベントログ）のみに一元化されています。
- `replayEvents` 関数が、初期カタログデータに対してイベントログを逐次適用（畳み込み）し、最新の `AppState` を出力します。
- `addToCart` や `checkout` などのコマンドは、まず現在の状態をリプレイして検証し、成功・失敗イベントをストアにプッシュするだけで、状態を直接変更しません。

## 自分なりの消化・考察 (Personal Insights)
- **理解のポイント:** 状態遷移の履歴が完全に残るため、「現在の状態」だけでなく「どうしてその状態に至ったか」の監査やデバッグが極めて容易になります。バグが発生した時点のイベントストリームがあれば、バグを100%再現可能です。
- **難しかった点 / 気づき:** 状態の変更をシミュレートする際、Projection（リプレイ処理）にのみデータ変更のロジックが集中するため、コマンド処理側が非常にスッキリする反面、頻繁なリプレイによるパフォーマンス低下（実務ではスナップショット等で対応する）への配慮が必要です。
- **実務への応用:** 金融取引システムや共同編集ツール（Google Docs等）、不変性が重視されるドメイン駆動設計（DDD）におけるCQRSパターンアーキテクチャに直接応用できます。
