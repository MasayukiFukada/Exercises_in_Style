# 30. Dataspaces (データスペース)

## 制約 (Constraints)
*   **直接通信の禁止:**
    スレッドやコンポーネントが、特定の相手（アクターなど）へ直接的な宛先を指定してメッセージやデータを送信することを禁止します。
*   **共有データスペース (TupleSpace) の利用:**
    すべてのやり取りは、全員が共有する「データスペース」に対して行われます。情報は「タプル（値の配列 / データ構造）」として配置（`write`）されます。
*   **パターンマッチングによる取得 (Lindaスタイル):**
    各自律ワーカーは、自身が処理できるパターン（例: `['catalog', '*', 'check_stock', '*']`）をスペースから照合して取得（`take`）し、非同期に処理して、再びスペースへ戻します。`take` されたデータはスペースから消去され、競合が自動解決されます。

---

## コードの特徴・解説
今回の実装では、パターン照合による待機・取得ができる `TupleSpace` クラスを自作し、カタログワーカーとカートワーカーを並行稼働させています。

*   **タプルスペースのパターンマッチ:**
    `take` はスペース内に合致するタプルがあれば即座に取り出して返し、なければ `Promise` を使ってワーカーの実行を一時中断し、タプルが `write` されるのを待機します。
    ```typescript
    class TupleSpace {
      private space: any[][] = [];
      // パターンが一致したら取り出し、削除する
      take(pattern: any[]): Promise<any[]>;
      write(tuple: any[]): void;
    }
    ```
*   **完全に疎結合なワーカー:**
    `catalogWorker` は `catalog` タプルを監視し、`cartWorker` は `cart` タプルを監視します。互いのインスタンス参照は持たず、呼び出し関係もタプルの受け渡し（`replyToken` による返信）によってのみ記述されています。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**時間と空間（宛先）の完全な非結合化（Decoupling）**です。
情報を送る側は「誰が処理するか」も「いつ処理されるか」も知る必要がなく、ただ共有のスペースにタスクを放り込むだけです。受ける側も、自分のタイミングで仕事を引き抜きます。これは、分散並行システムにおける究極の疎結合化モデル（Linda / TupleSpace パラダイム）です。

### 難しかった点 / 気づき
*   **状態共有の代替とデータ競合の回避:**
    カート追加の際、カタログの在庫情報が必要になります。アクターモデルでは直接相手にメッセージを送りますが、データスペースでは「在庫確認タプル」を書き込み、カタログワーカーが返した「在庫結果タプル」を `take` するという往復が必要です。この時、複数の要求が混ざらないよう、固有の `replyToken`（相関ID）をタプルに埋め込んで正確にパターンマッチングさせる必要があります。

### 実務への応用
*   **メッセージキュー / ジョブキュー (Celery, Sidekiq, BullMQ):**
    Webサーバー（プロデューサー）が Redis や RabbitMQ などのキュー（データスペース）に「メール送信タスク」を `write` し、裏側の複数の Worker（コンシューマー）がそれを `take` して非同期に順次処理する構成は、まさにこのスタイルです。
*   **分散並列コンピューティング (Apache Spark, MapReduceのバッファ):**
    大量のタスクをクラスター内の多数の計算ノードに分散させて処理を配分する際の基礎構造となっています。
