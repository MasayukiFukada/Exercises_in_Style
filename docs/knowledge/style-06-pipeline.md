# 06. Pipeline (パイプライン)

## 制約 (Constraints)
*   **副作用のない純粋関数 (Pure Functions):**
    各手続きはグローバル変数の書き換えや、引数で渡されたオブジェクトの直接破壊（破壊的変更）を行ってはなりません。
*   **イミュータブル (不変) な状態の伝播:**
    状態の更新が必要な場合は、既存の状態を直接変更せず、常に複製した新しい状態オブジェクト（`AppState`）を作成して返します。
*   **関数合成（パイプライン）による制御フロー:**
    手続きの途中結果をローカル変数に代入して一時保持することを避けます。ある純粋関数の出力を次の純粋関数の入力として一本のチェーンのように繋ぎ、全体のフローを制御します。

---

## コードの特徴・解説
今回の実装では、関数型プログラミング（FP）の「イミュータブルな状態遷移」と「関数合成」を主軸としています。

*   **状態オブジェクトの不変性 (`readonly`):**
    商品カタログおよびカート情報は、すべて読み取り専用（`readonly`）に設定され、直接のプロパティ変更がコンパイラレベルで防止されています。
*   **高階関数（カリー化）による関数部分適用:**
    `addToCart('PRD-02', 1)` は、引数を事前にバインドし、`AppState -> AppState` という「状態を受け取って新しい状態を返す純粋関数」を動的に生成して返します。
*   **`pipe` ヘルパーによる合成関数の実行:**
    複数の純粋関数を一本のパイプラインに連結する `pipe` 関数を用意し、テストシナリオの全工程を一気にデータが流れていく仕組みにしています。
    ```typescript
    const runShoppingSimulation = pipe(
      printCatalog,
      addToCart('PRD-02', 1),
      addToCart('PRD-03', 1),
      checkout,
      printCatalog
    );
    ```

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**プログラムを「状態の変化の歴史（データフロー）」としてモデル化すること**です。
状態を「変数」という器の中に保持し、それをインプレースで書き換えていく命令型スタイルとは異なり、データが関数というトンネルを通過するたびに新しい形へと変形（Transform）していく様子を表しています。

### 難しかった点 / 気づき
*   **シャローコピーとデータ更新のコスト:**
    商品を追加するたびに `[...state.cart]` や `{ ...prod }` を使って新しいオブジェクトを生成します。メモリ消費やパフォーマンスの観点ではインプレースの書き換えよりコストがかかりますが、状態の変化を追跡しやすく、バグを防ぐ上での心理的安全性が非常に高いです。
*   **エラー処理の伝播:**
    パイプラインは途中で処理を「中断」させることが難しいため（エラーが発生しても、パイプラインの最後までデータを流す必要がある）、エラーが発生した場合は「エラーを含む新しい状態オブジェクト」を生成して後続に託すという設計が必要になります。今回は簡単のためにエラー時は状態を変更せずにそのまま次にパスする設計にしました。

### 実務への応用
*   **Redux / 状態管理ライブラリ (Reactなど):**
    Reactの `useState` や Redux の `Reducer` パターンは、このイミュータブルな状態更新（`(state, action) => newState`）そのものです。このスタイルはモダンなフロントエンド開発の基礎技術と完全に合致しています。
*   **関数型プログラミング言語 (Haskell, Elixir, Scalaなど):**
    これらの言語で標準的に使われる「パイプ演算子（`|>`）」の概念を体現しています。
*   **スレッドセーフ（並行処理の安全性）:**
    共有の可変状態（Shared Mutable State）を持たないため、マルチスレッド環境においてデータ競合（Data Race）やデッドロックを気にする必要が全くなく、並行処理と極めて親和性が高いです。
