# 04. Monolith (モノリス)

## 制約 (Constraints)
*   **名前付き関数の完全な禁止:**
    プログラム内で名前付き手続きや関数（サブルーチン）を定義して処理を分割することは禁止されています。
*   **単一のコードブロック:**
    すべての処理、状態管理、テストシナリオの実行が、上から下へと流れる1つのグローバルなシーケンシャルブロックとして書かれます。
*   **コピペによる再利用 (インライン展開):**
    カート追加などの共通のビジネスロジックは、再利用可能な関数にするのではなく、必要な箇所に毎回コードをインライン展開（コピー＆ペースト）して記述します。

---

## コードの特徴・解説
今回のショッピングカート実装では、関数の定義を一切行っていません。

*   **同一ロジックの繰り返し:**
    テストストーリーにおける「マウスを追加」「キーボードを追加（失敗）」「ノートPCを追加（失敗）」「ノートPCを追加（成功）」の4回のカート追加処理は、共通の「カタログ検索 $\rightarrow$ 在庫チェック $\rightarrow$ カート追加」のロジックが4回コピーされ、ブロック `{ ... }` を用いてローカルスコープを分けつつインライン展開されています。
*   **直線のコントロールフロー:**
    プログラム全体の開始から、商品カタログ表示、複数回のカート追加、チェックアウトによる在庫減算、そして最終的なカタログ表示までが、分岐やループを含みつつも直線的なステップとしてシーケンシャルに処理されます。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**関数という「手続きの抽象化」がない世界がいかに過酷か、そして関数がどれほど強力なプログラムの組織化手段であるかを身をもって理解すること**です。
構造化プログラミング以前の初期のアセンブラやBASICのような、コードを小さなまとまりに分割する手段を持たないシステムを再現しています。

### 難しかった点 / 気づき
*   **DRY（Don't Repeat Yourself）原則の破壊に伴う保守コスト:**
    全く同じロジックを4箇所にコピペしたため、もし「在庫チェックのルールを変更したい（例: 在庫不足時はエラーにせず限界まで追加する）」となった場合、コピーした4箇所のコードすべてを漏れなく修正しなければなりません。修正漏れによるバグの発生確率が跳ね上がるのを実感し、**関数の「変更箇所の局所化（単一責任の原則）」がいかに重要か**が痛感できました。
*   **認知負荷の増大:**
    コードが構造的に分割されていないため、全体で150行程度の短いコードでもスクロールしながら処理の流れを追いかけるのが非常に困難になります。関数の境界線（名前とシグネチャ）が、読者にとっての強力な「目次」の役割を果たしていることがよく分かります。

### 実務への応用
*   **インライン展開 (Inlining) による最適化の理解:**
    コンパイラは実行速度向上のため、関数呼び出しのオーバーヘッドを避けるべく、自動的に関数をインライン展開（このモノリススタイルのような状態に変換）することがあります。このコード表現はそのコンパイラ最適化の内部結果を手動で再現していると言えます。
*   **リファクタリングの動機付け:**
    レガシーコードにおける「巨大すぎるメソッド（メソッドの肥大化）」に出会った際、それをどうやって関数に抽出すべきか（Extract Method）の判断力と、その重要性に対する確信が深まります。
