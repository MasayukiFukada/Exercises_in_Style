# 22. Tantrum (かんしゃく / 契約による設計)

## 制約 (Constraints)
*   **契約による設計 (Design by Contract - DbC):**
    各処理の前後で状態の正当性を検証する「契約」を取り決めます。
    *   **事前条件 (Preconditions)**: メソッド呼び出し前に呼び出し側が満たすべき条件。
    *   **事後条件 (Postconditions)**: メソッド完了後に呼び出された側が保証する条件。
    *   **不変条件 (Invariants)**: オブジェクトの生存期間中、常に維持されるべき状態の条件。
*   **即時例外スロー (かんしゃく):**
    いずれかの条件が破られた場合、決して処理を曖昧に進めてはならず、即座に例外（Assertion Error）をスローしてプロセスを緊急停止させます。

---

## コードの特徴・解説
今回の実装では、`assert()` 関数を用いて、カート追加、在庫減算、チェックアウト処理におけるすべてのステップで事前・事後・不変条件を厳しくチェックしています。

*   **契約の定義例 (Product.decreaseStock):**
    ```typescript
    decreaseStock(qty: number): void {
      // 事前条件: 数量が正であること、在庫が十分あること
      assert(qty > 0, "...");
      assert(this.stock >= qty, "...");

      const originalStock = this.stock;
      this.stock -= qty;

      // 事後条件: 在庫が正しく引かれ、不整合（マイナス在庫など）が起きていないこと
      assert(this.stock === originalStock - qty, "...");
      assert(this.stock >= 0, "...");
    }
    ```
*   **実行時アサーションのキャッチ:**
    例外発生によって処理が即座に中断されるため、テストストーリー側では `try-catch` を用いて、例外（かんしゃく）が発生したことをハンドリングしつつ処理を次に進めています。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**「間違った状態で処理を進めるくらいなら、その場でクラッシュさせた方がはるかにマシである」**というフェイルファスト (Fail-Fast) の思想です。
不整合なデータを無理に処理し続けると、後からデータ破損などの深刻な二次災害を引き起こすため、問題の根源で即座に検知してエラーを投げます。

### 難しかった点 / 気づき
*   **エラーの責任分界点の明確化:**
    事前条件の違反は「呼び出し側（Caller）」のバグであり、事後条件の違反は「呼び出された側（Callee）」のバグを意味します。アサーションメッセージによって、どちらの責任で契約が破られたかが一目瞭然になるため、デバッグが極めて容易になります。
*   **コードの肥大化:**
    ロジックそのものよりも `assert` による検証コードの方が多くなりがちです。実務ではプロダクション環境でこれらのアサーションをコンパイル時に取り除く仕組み（アサーションストリップ）が併用されることもあります。

### 実務への応用
*   **ドメインモデルでのバリデーション:**
    DDDにおけるエンティティのコンストラクタで、ドメインのルール（例: 「注文合計金額は負になり得ない」など）を満たさない値オブジェクトの生成を `assert` や例外で弾くガード節として使われます。
*   **静的解析・型検証との相乗効果:**
    Rustの `assert!` マクロや、C++の `static_assert`、Javaの `assert` キーワードなど、システムプログラミングにおける安全性確保の必須テクニックです。
