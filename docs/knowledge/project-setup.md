# プロジェクト基本方針・共通仕様 (Project Setup & Specification)

本プロジェクトにおける開発言語、実装対象となるシステムの仕様、および各スタイル共通の動作確認用シナリオを定義します。

---

## 1. 使用言語の選定
本プロジェクトでは **TypeScript (Node.js)** を使用言語とします。

### 選定理由
*   **マルチパラダイム対応:** クラスベースのオブジェクト指向、関数型プログラミング（高階関数、パイプライン）、アクター風のメッセージングなどを強力にサポートしています。
*   **高度な型システム:** インターフェースや型定義、および型安全性の比較（静的型付け vs 動的型付け）が非常に明確に表現できます。
*   **低レベル操作のシミュレーション:** `Good Old Times` などのメモリ制限スタイルにおいて、型付き配列（TypedArray）などを利用してメモリ空間をエミュレートすることが容易です。

---

## 2. 対象システム：ECショッピングカート
単語の出現頻度カウントの代わりに、ビジネスロジックや状態遷移の比較がしやすい**「ECサイトのショッピングカート」**を実装対象とします。

### コアエンティティ（データモデル）
1.  **商品 (Product):** ID、名前、価格（単価）、在庫数。
2.  **カートアイテム (CartItem):** 商品、数量。
3.  **ショッピングカート (ShoppingCart):** カートアイテムのリスト、合計金額、割引適用状態。

### 基本機能
1.  **商品カタログの参照:** システム内の利用可能な商品一覧を取得する。
2.  **カートへの追加 (Add to Cart):**
    *   指定した商品を数量指定でカートに追加する。
    *   在庫数を超える追加要求があった場合はエラーとする（または在庫の上限まで追加する）。
3.  **割引計算 (Discount):**
    *   「合計金額が 3,000円 以上の場合は 10% 割引」などの簡単な割引ルールを適用する。
4.  **チェックアウト (Checkout):**
    *   注文を確定し、商品の在庫を減算する。
    *   レシート（購入明細と最終合計金額）を出力する。

---

## 3. 動作確認用テストストーリー (共通シナリオ)
各スタイルの実装が正しく動作しているかを比較するため、すべてのスタイルで共通のシナリオを実行し、**コンソール（標準出力）に結果ログを出力**します。
UIやデータベースといった外部連携は行わず、コンソールへの出力のみで動作確認を行います。

### 初期データ (商品カタログ)
*   `PRD-01`: ノートPC / 100,000円 / 在庫 2個
*   `PRD-02`: マウス / 3,000円 / 在庫 5個
*   `PRD-03`: キーボード / 5,000円 / 在庫 0個（品切れ）

### シナリオ実行フロー
1.  **商品一覧の表示:** カタログからすべての商品と在庫状況を出力する。
2.  **正常な追加:** `PRD-02` (マウス) を 1個 カートに追加する。
3.  **在庫切れの追加試行:** `PRD-03` (キーボード) を 1個 追加しようとする $\rightarrow$ 在庫切れのためエラーメッセージを表示し、カートには追加されない。
4.  **在庫超えの追加試行:** `PRD-01` (ノートPC) を 3個 追加しようとする $\rightarrow$ 在庫不足（最大2個）のためエラーメッセージを表示し、カートには追加されない。
5.  **複数商品の追加:** `PRD-01` (ノートPC) を 1個 追加する。
6.  **カート状態の確認:** カート内の明細と、割引前の合計金額（103,000円）を表示する。
7.  **チェックアウト:** 
    *   3,000円以上のため10%割引を適用する（103,000円 $\rightarrow$ 92,700円）。
    *   在庫を減算する（ノートPCの在庫は残り1個、マウスの在庫は残り4個になる）。
    *   最終的なレシート（購入明細、割引額、支払合計金額）を出力する。
8.  **事後の在庫確認:** カタログの在庫が正しく減算されていることを確認する。

---

## 4. プログラミングスタイルの適用イメージ（例）
このショッピングカートの仕様を、各スタイルの制約に当てはめて実装します。

*   **01. Good Old Times:** クラスやオブジェクトは使わず、1つの巨大なグローバル配列（メモリ空間）を用意し、特定のアドレス（例: インデックス 0〜10 は商品データ、100〜110 はカートデータ）に数値を書き込んで状態を管理する。
*   **06. Pipeline:** カタログ、カート、割引、在庫減算の処理をそれぞれ独立した純粋関数とし、カート状態オブジェクトをパイプラインで流して最終的なレシートを得る。
*   **12. Letterbox:** 商品、カート、在庫管理システムを個別のオブジェクトとし、互いに「add_item」「checkout」といったメッセージを送信し合って協調動作する。
