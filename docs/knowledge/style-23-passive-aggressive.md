# 23. Passive-Aggressive (受動攻撃的 / 例外ハンドリング)

## 制約 (Constraints)
*   **下位でのエラー回復の禁止:**
    各ドメインクラス（`Product`, `ShoppingCart`, `CheckoutService` など）や補助メソッドの中で、エラーをその場で解決しようとしたり、`try-catch` を記述して回復処理を行うことを禁止します。
*   **例外の受動的スロー:**
    エラーを検知した際は、単に `throw new Error(...)` で例外を上位へ投げつけます（受動攻撃的）。
*   **最上位での集中エラー管理:**
    すべての例外のキャッチおよびエラーメッセージのログ出力は、プログラムの最外周（メインループやシナリオ進行部などの最上位エントリーポイント）で一括して行います。

---

## コードの特徴・解説
今回の実装では、ドメインクラス内から `try-catch` を完全に排除し、メインシナリオを実行するループ部分で一括して例外をトラップする構成にしています。

*   **ドメイン内のエラーの丸投げ:**
    条件を満たさない場合は、自身で警告を出すことなく、即座に例外をスローします。
    ```typescript
    class ShoppingCart {
      addItem(product: Product | undefined, qty: number): void {
        if (!product) throw new Error("...");
        if (product.stock === 0) throw new Error("...");
        // ...
      }
    }
    ```
*   **最上位での一括キャッチ:**
    シナリオの各工程を関数の配列として定義し、メインループがそれらを `try-catch` で包みながら順次実行します。
    ```typescript
    for (const step of steps) {
      try {
        step();
      } catch (error: any) {
        // ここで初めてエラー出力を実行
        console.log(`エラー: ${error.message}`);
      }
    }
    ```

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**下位コンポーネントが余計な責任（エラーハンドリング）を持たず、関心事をコアのビジネスロジックのみに集中させること**です。
下位で個別に `try-catch` を記述してその都度エラーをログに吐いて元の値を返すような「中途半端な回復」を行うと、上位側でエラーが起きたこと自体を検知できなくなります。エラーの通知を例外バブルアップに委ねることで、コードが極めて簡潔になります。

### 難しかった点 / 気づき
*   **スタックの維持と処理の継続:**
    通常のプログラムにおいて、例外を最上位でキャッチするとその時点でプログラム全体の実行が終了します。しかし、今回のショッピングカートのシナリオのように「途中でエラーが起きても、以降のステップ（追加やチェックアウト）を継続したい」という要件がある場合、ステップごとに区切られたトランザクションの境界を最上位レイヤー側で定義し、ループ内で個別キャッチするような制御構造を設計する必要があります。

### 実務への応用
*   **Webアプリケーションフレームワークのミドルウェア:**
    Express, Koa, NestJS などのWebサーバー設計では、各ルーティングハンドラの中では `try-catch` を書かずに、データベースエラーや検証エラーをそのまま `throw` (または `next(err)`) し、グローバルな例外フィルター（エラーハンドリングミドルウェア）が最上位で一括キャッチして適切な HTTP ステータス（500 や 400）に変換して返します。
*   **集中化されたエラーロギング:**
    Sentry や Bugsnag などのエラー監視ツールと連携する場合、エラーキャッチ箇所を一箇所に集約することで、漏れなくすべての例外スタックトレースを収集・報告する体制が整います。
