# 46. Reactive Streams (リアクティブストリーム)

## 制約 (Constraints)
- システムへの命令（カタログ表示、カート追加、チェックアウトなど）は、時間とともに流れる「イベントストリーム (Observable)」として表現される。
- 状態の直接更新や変数での引き回しを排し、イベントストリームに対する宣言的な変換オペレータ（`map`, `filter`, `scan`）の適用のみでビジネスロジックを構成する。
- システムの状態遷移は、`action$` ストリームから `scan` オペレータを用いて状態オブジェクト（`AppState`）に畳み込むことで自動的に伝播・再現される。
- コンソール出力やデータの最終的な反映などの副作用は、最終的なストリームの購読（`subscribe`）によってのみ実行される。

## コードの特徴・解説
- アクションを順次発行する `action$` (Subject) を起点とし、それを `scan` することで過去のイベントを畳み込んだ状態（`AppState`）を流す `state$` ストリームを構築します。
- `state$.subscribe(...)` によってのみ、コンソール表示やレシート印刷といった副作用を実行し、ビジネスロジック（`scan` の内部）と副作用（出力）を明確に分離しています。

## 自分なりの消化・考察 (Personal Insights)
- **理解のポイント:** `27. Spreadsheet` が「セルのデータ依存関係（プル/プッシュの双方向バインディング）」に焦点を当てていたのに対し、`46. Reactive Streams` は「時間とともに流れる非同期イベントストリームの一方向フロー」に焦点を当てています。手続き的呼び出しではなく、「値の変化の伝播」をストリームの連結として定義する点がリアクティブの核心です。
- **難しかった点 / 気づき:** 外部ライブラリ（RxJSなど）を使用せず、簡易的な `Observable` や `Subject` を TypeScript でフルスクラッチで記述しつつ、`scan` のシード値やパイプラインの伝播を型安全に設計する点が良いパズルになりました。
- **実務への応用:** Webアプリケーションにおけるイベント処理（クリックストリーム、リアルタイムチャットメッセージ、WebSockets）、およびバックエンドでのデータインジェストパイプライン（KafkaやReactive Streamsを利用したリアクティブシステム設計）において、この考え方が直接活かされます。
