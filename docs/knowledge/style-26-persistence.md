# 26. Persistence (永続化)

## 制約 (Constraints)
*   **メモリ上での状態保持の禁止:**
    アプリケーションの状態（このプロジェクトでは商品カタログデータやカートのアイテムリスト）を、プログラムのプロセス内（メモリ上のグローバル変数や永続オブジェクト）に保持することを一切禁止します。
*   **ファイル/DBへの即時読み書き:**
    すべてのデータの参照（クエリ）と更新（トランザクション）は、その都度外部ファイル（またはデータベース）から読み出し、処理後にファイルへ再書き込みして状態を同期させます。
*   **ステートレスなプログラムプロセス:**
    各サービスのメソッド実行完了後、メモリ上の関連する変数や状態の参照はすべて破棄され、次の操作時には再びファイルからフレッシュに読み込み直します。

---

## コードの特徴・解説
今回の実装では、`database.json` というJSON形式のファイルを簡易リレーショナルデータベースとして使用しています。

*   **データロードとセーブの徹底:**
    すべての操作は `readDB` で最新状態を外部ファイルから読み出し、更新後は `writeDB` でファイルにコミットします。
    ```typescript
    class ShoppingCartService {
      addItem(productId: string, qty: number): void {
        const db = readDB(); // クエリ実行
        // ... カート状態の変更 ...
        writeDB(db); // コミット
      }
    }
    ```
*   **状態共有の排除:**
    `ShoppingCartService` や `CheckoutService` は、内部状態を指し示すインスタンスフィールドを一切持たず、純粋にDBファイルを操作する機能クラス（ステートレスサービス）として設計されています。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**プログラムと状態の完全な分離**です。
メモリ上に状態を持たないため、プログラムが急にクラッシュしたり、プロセスが強制終了されたりしても、直前の状態は完全に外部ファイルへ保護されています。また、これは Webにおける RESTful API やデータベーストランザクション、ステートレスサーバー設計の極めて重要な基本モデルです。

### 難しかった点 / 気づき
*   **I/Oのボトルネックと整合性:**
    各アクションごとにファイル全体の `readFileSync` / `writeFileSync` が発生するため、実行パフォーマンスは著しく低下します。実際のシステム開発では、このボトルネックを防ぐために、キャッシュ（メモリ）とデータベースの適切な同期タイミングや、変更行（差分）のみを更新する仕組み（RDBのトランザクションログやインデックス）が必要になります。

### 実務への応用
*   **データベース連携するアプリケーション:**
    ほとんどの現代のエンタープライズWebアプリケーションは、サーバーメモリ上にユーザーカートや商品データを置かず、操作の度に PostgreSQL や MySQL, Redis などのデータベースサーバーに SQL クエリを投げてデータを操作します。
*   **コンテナ化（Docker）やサーバーレス（AWS Lambda）:**
    サーバーレスアーキテクチャでは実行コンテキストがリクエストごとに消滅（破棄）するため、あらゆるデータをDynamoDBなどの外部ストレージに永続化させ、プロセスは完全にステートレスにするという Persistence スタイルが標準設計となります。
