# 45. Logic Programming (論理・制約プログラミング)

## 制約 (Constraints)
- 手続き的な制御フロー（ビジネスロジック層での `if-else` による命令的な分岐や状態の直接書き換え）を禁止する。
- すべての情報は「データベース (Knowledge Base)」に「事実 (Facts)」として登録する。
- 判定や計算は、データベースに対する「クエリ (Query)」を実行し、登録された「事実」と「規則 (Rules)」の照合によって答えを導き出す。
- 状態の変化は、事実の登録 (Assert) と削除 (Retract) によって表現する。

## コードの特徴・解説
- 商品情報（`product`）やカート情報（`cart`）はすべて `KnowledgeBase` クラスの中に `Fact` 型のデータとして格納されます。
- 「カートに追加できるか？」というビジネスロジックは、`evaluateAddToCartRule` 関数が知識ベースから商品事実やカート事実をクエリし、ルールに合致するかを評価する宣言的な規則として定義されています。
- 追加に成功すると、古いカート事実を `retract`（削除）し、加算した新しい事実を `assert`（登録）します。

## 自分なりの消化・考察 (Personal Insights)
- **理解のポイント:** 「どのように処理を順次実行するか」ではなく、「何が正しい状態であるか（ルール）」をデータベースに対して定義し、状態更新自体もデータベースの操作に落とし込むというProlog的なパラダイムです。
- **難しかった点 / 気づき:** 手続き型言語（TypeScript）の持つオブジェクトの参照や制御構造に頼らず、一歩引いて「データベース上の事実の書き換え」として考えることで、ビジネスロジックを極めてシンプルなルール記述に分離できました。
- **実務への応用:** 大規模な割引計算、ポリシーチェック、アクセス制御、ルールベースのバリデーションエンジンなど、複雑なビジネスルールをデータとして外部定義し、実行時に動的にルール評価するエンジン（Droolsやミニルールソルバー）を構築する際に活かせます。
