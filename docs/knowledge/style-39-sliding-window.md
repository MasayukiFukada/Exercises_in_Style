# 39. Sliding Window (スライディングウィンドウ)

## 制約 (Constraints)
- **フラットなデータ配列の利用**: データ（在庫数、カート内数量、価格など）は個別のオブジェクトや複雑なデータ構造ではなく、1次元のフラットな数値シーケンス（配列）として表現する。
- **ループや個別if-else判定の排除**: カートの数量判定や割引適用可否などのビジネスルール判定において、明示的な `for/while` ループや `if-else` による個別要素チェックを行わない。
- **スライディングウィンドウの適用**: 一定の窓幅（Window Size）と歩幅（Stride）で部分配列を切り出すためのインデックスマップ（インデックス行列）をあらかじめ生成する。
- **一括演算と活性化関数の適用**: 抽出されたウィンドウ行列に対し、重みベクトルとの積（線形結合）や活性化関数（ステップ関数など）を一括適用して状態判定を行う。

## コードの特徴・解説
- `getSlidingWindowIndices` 関数で、データの長さに応じたウィンドウインデックスの二次元配列を生成します。例えば、[在庫, カート, 要求] という3つの特徴量を1セットとして処理するため、ウィンドウサイズ `3`、ストライド `3` のインデックス（`[[0, 1, 2], [3, 4, 5], ...]`）を作ります。
- `applyWindow` によって、フラットなデータ配列からインデックス行列を用いてデータを窓単位で抽出（スライディングウィンドウ処理）します。
- `checkStockAvailability` では、抽出された各窓に対して重み `[1, -1, -1]`（現在の在庫数 - カート内数量 - 要求数量）とバイアス `0` で内積を計算し、結果にステップ関数を適用することで、ループや個別の if 条件を使用せずに全商品の在庫判定を一括して予測・決定しています。
- 小計の計算や割引適用の判定でも、同様に価格と数量をウィンドウサイズ `2`, ストライド `2` で切り出して一括計算を行い、合計金額に重み `[1]` とバイアス `-3000` を適用する単層ネットワークで割引適用を決定しています。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
*   **CNNの基礎となる考え方:** ループや個別オブジェクトへの条件分岐を排除し、インデックスマップによってデータを窓幅（Window Size）と歩幅（Stride）で切り出し、一括して行列演算を適用するアプローチです。
*   **バッチ処理の効率化:** 明示的なループを書くのではなく、データを特定のインデックスで切り出して行列演算に落とし込むことで、GPUやベクトル演算ユニットによる並列処理（バッチ処理）を行いやすい構造になっています。

### 難しかった点 / 気づき
*   **ビジネスロジックの変換:** オブジェクト指向や手続き型の `if (stock < cart + req)` のような条件式を、フラットな配列とスライディングウィンドウによる重み演算に変換するパズル的な再設計が必要でした。
*   **隣接関係へのマッピング:** ストライドとウィンドウサイズを適切に設定することで、オブジェクトの属性値を「配列上のインデックス選択と隣接関係」に翻訳できる点は非常に面白いアプローチです。

### 実務への応用
*   **時系列分析・ログ監視:** 一定の時間枠（スライディングウィンドウ）ごとにログやメトリクスを切り出して異常検知や移動平均を一括計算するタスクに最適です。
*   **行動ログのパターン検出:** Webアプリケーションにおけるユーザー操作のタイムラインから、特定の操作シーケンスパターンをバッチ処理で検出するようなフィルターロジックに応用できます。
