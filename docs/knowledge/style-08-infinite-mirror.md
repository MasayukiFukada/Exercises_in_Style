# 08. Infinite Mirror (無限の鏡 - 再帰)

## 制約 (Constraints)
*   **反復制御構造の禁止:**
    `for`, `while`, `do-while` などの命令的なイテレーション構文を一切使用してはなりません。
*   **再帰による繰り返し表現:**
    配列の走査、特定のデータの検索、集計処理といったすべての反復処理は、関数が自分自身を呼び出す「再帰（Recursion）」によって記述する必要があります。
*   **明確なベースケース（終了条件）の定義:**
    無限ループ（無限再帰）によるスタックオーバーフローを防ぐため、各再帰関数には明確な終了条件を定義しなければなりません。

---

## コードの特徴・解説
今回の実装では、あらゆる走査ロジックを再帰関数へ置き換えています。

*   **インデックスを用いた配列走査:**
    `printCatalogRecursive(products, index)` のように、走査位置を示す `index` を引数として持ち、`index >= products.length` というベースケースに達するまで自身を呼び出します。
*   **再帰による検索:**
    商品やカート内アイテムの検索（`findProductRecursive` / `findCartItemRecursive`）も、対象が見つかるか配列の末尾に達するまで再帰的に評価されます。
*   **再帰での「出力と状態集計」の同時実行:**
    チェックアウト処理の `checkoutItemsRecursive` は、レシートの明細出力を再帰で順次行いながら、コールスタックの戻り値の積み重ねによって「小計の合計金額」を計算して呼び出し元へ戻します。
    ```typescript
    return lineTotal + checkoutItemsRecursive(products, cart, index + 1);
    ```

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**状態の反復を「時間の経過（ループ）」ではなく「構造のネスト（再帰スタック）」として表現すること**です。
命令型プログラミングでは、ループを用いて「状態を順次書き換えていく」のに対し、再帰型（関数型）プログラミングでは「現在の入力に対する結果 ＝ 最初の要素の処理 ＋ 残りの要素に対する同じ処理の結果」というように、問題を数学的な「帰納法」で定義します。

### 難しかった点 / 気づき
*   **頭の切り替え（帰納的思考）:**
    普段、無意識に `for` ループで書いている処理をすべて再帰に分解する際、少し思考の切り替えが必要です。「この処理の最後の状態はどうなるか（ベースケース）」を常に最初に定義し、それを起点に処理を巻き戻していく感覚は、パズルを解くような知的な面白さがあります。
*   **スタックオーバーフローのリスク:**
    TypeScript (JavaScript) は、一部の環境を除いて標準で「末尾呼び出し最適化 (Tail Call Optimization: TCO)」が完全に有効化されているわけではないため、大規模なデータ（数万件のカートなど）に対してこのスタイルを適用すると、コールスタックが限界に達してクラッシュ（Stack Overflow）します。実務ではデータの規模に対する考慮が不可欠です。

### 実務への応用
*   **木構造 (Tree) やグラフ (Graph) などの階層データ処理:**
    ファイルシステム（ディレクトリ探索）、DOMツリー、JSONオブジェクトの深さ優先探索など、再帰的構造を持つデータの走査は、ループよりも再帰で書く方が圧倒的に直感的に記述できます。
*   **分割統治法 (Divide and Conquer):**
    マージソートやクイックソート、二分探索など、アルゴリズムの高速化や効率的な分割処理を設計するための必須の考え方です。
