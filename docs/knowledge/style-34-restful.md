# 34. RESTful (ステートレス)

## 制約 (Constraints)
*   **クライアント/サーバー分離 (Client-Server):**
    プログラムを要求を投げる側（クライアント）と、要求を処理する側（サーバー）に明確に分離します。両者の対話は HTTP リクエストとレスポンスの形式（エミュレート）のみで行われます。
*   **ステートレス性 (Statelessness):**
    サーバーはクライアントのセッション状態（現在どのユーザーがカートに何を入れているか）をデータベースやメモリに一切保持しません。
*   **状態のクライアント引き回し:**
    クライアントは、自身の状態（カート内の全アイテム）を常にローカルで保持し、サーバーへ操作要求（追加やチェックアウト）を送る際は、毎回現在のカート情報全体をリクエストボディに含めて送信します。
*   **リソース指向とインターフェース:**
    操作対象は一意のパス（URI風、例: `/cart/items`）で指定され、標準的なメソッド（GET, POST）によって決定されます。

## コードの特徴・解説
今回の実装では、`RESTfulServer` クラスを定義し、仮想的なネットワークリクエスト `handleRequest` を介してステートレス対話をエミュレートしています。

*   **ステートレスサーバー:**
    サーバーのメモリにある状態は商品マスタ（データベース）のみです。カートに追加する要求を受け取った際、サーバーは「カートの状態」を保持していないため、クライアントから送られてきたカートデータにアイテムを追加し、新しいカートデータをレスポンスとしてクライアントに送り返します。
    ```typescript
    // POST /cart/items
    const { cart, productId, qty } = body;
    // ... 在庫検証 ...
    // クライアントのカートデータを変形して送り返す
    return { status: 200, body: { cart: updatedCart } };
    ```
*   **クライアントの状態管理:**
    クライアントは `let clientCart` という変数にカート状態を維持し、追加やチェックアウト時には常にそれを送信し、レスポンスから返される最新のカート状態を保存（再同期）します。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**サーバーの水平スケール（拡張性）の最大化と耐障害性の確保**にあります。
もしサーバーがメモリ上でユーザーセッション（カート）を保持している場合、サーバーが1台ダウンするとユーザーのカートの中身が消えてしまいます。また、ユーザーは常に同じサーバーにアクセスしなければなりません（スティッキーセッション）。サーバーをステートレスにすることで、クライアントは任意のサーバーにリクエストを送信できるようになり、システム全体の可用性が劇的に向上します。

### 難しかった点 / 気づき
*   **データ転送量の増加:**
    カートに商品を追加するだけで、カート全体のリストをサーバーとクライアントの間で毎度往復させる必要があります。これはカートの規模が大きくなると通信オーバーヘッド（帯域消費）の原因となります。実務では、セッション状態を Redis などの高速な外部共有インメモリキャッシュに逃がすことで、サーバーのステートレス性とデータ軽量化を両立させることが一般的です。

### 実務への応用
*   **REST API / HTTP Webサービス:**
    現代のWeb API（JSON over HTTP）の標準的な設計ガイドラインです。認証情報の受け渡しにおいて、サーバーセッションを使わずに毎リクエストで JWT（JSON Web Token）などの認証トークンを送るのも、このステートレス設計に立脚しています。
*   **SPA (Single Page Application) とサーバー開発:**
    ReactやVueなどで記述されたフロントエンド側がアプリケーション状態を管理し、バックエンドは純粋なAPI（データのCRUD）としてステートレスに動かす、現代の標準システム構成です。
