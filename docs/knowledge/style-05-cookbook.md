# 05. Cookbook (クックブック)

## 制約 (Constraints)
*   **手続きへの適切な分割 (構造化):**
    巨大な処理をそのまま記述せず、カタログ表示、カート追加、チェックアウトのように、明確な単一の役割を持った手続き（関数・サブルーチン）に分解します。
*   **共有状態（グローバル変数）の最小化:**
    グローバルな共有状態をできる限り排除します。各サブルーチンは、必要とする状態データ（`catalog` や `cart`）を引数（パラメータ）として外部から明示的に受け取る必要があります。
*   **明示的なインターフェースの利用:**
    手続きの入出力は、引数（入力）および戻り値（出力）を介して行われます。

---

## コードの特徴・解説
今回の実装は、私たちが日常的に最も親しんでいる「構造化された手続き型プログラミング」です。

*   **状態のローカル化:**
    カタログやカートといった状態データは、グローバル空間ではなくメインのエントリポイント関数 `runShoppingSimulation` 内のローカル変数として定義されています。
*   **引数による明示的な依存性の引き回し:**
    `addToCart(products, currentCart, targetId, qty)` のように、各関数は処理対象のデータをパラメータとして受け取り、戻り値（エラーメッセージまたは成功フラグ）を返すだけで動作する、自己完結性の高いコードになっています。
*   **レシピ風のメインルーチン:**
    メインルーチンは、「カタログ表示」「追加」「追加」「チェックアウト」のように、レシピ本（Cookbook）の手順通りにサブルーチンを呼び出すことで、ドメインのストーリーを宣言的に表現しています。

## 自分なりの消化・考察 (Personal Insights)

### 理解のポイント
このスタイルの本質は、**プログラムを人間が理解しやすい適切な粒度の「部品（手続き）」に分割し、境界線を敷くこと**です。
スパゲッティコードになりがちな「04. Monolith」や、グローバル共有状態で頭痛の種となる「01. Good Old Times」の諸問題を解決し、チーム開発や保守性を劇的に向上させた歴史的マイルストーン（構造化プログラミングの時代）を反映しています。

### 難しかった点 / 気づき
*   **設計の心地よさと安心感:**
    これまでのスタイル（メモリ番地管理や、ひたすらインライン展開するコード）と比較して、圧倒的に「書きやすく、バグを特定しやすい」です。変数や関数のスコープが限定されているため、その関数を読んで修正する際の「局所的な認知負荷」が非常に低くなります。
*   **引き回し（パラメータ）の増大:**
    グローバル変数を使わない代償として、複数の関数に同じデータ（`catalog` と `cart`）を何度も渡し続けなければならない「引数の引き回し問題」が発生します。これがさらに大規模になると、「状態と手続きを一体化してカプセル化したい」という欲求が生まれ、後の「オブジェクト指向プログラミング（OOP）」へと繋がる流れが理解できます。

### 実務への応用
*   **DRY（Don't Repeat Yourself）と可読性の両立:**
    日常的なプログラミングにおいて、最初に意識すべき「関数の適切な抽出（Extract Method）」の基本原則そのものです。
*   **テストの容易性 (Testability):**
    各関数が引数を受け取って結果を返すクリーンなつくりであるため、ユニットテストを記述する際にモックなどを用意せず、モックデータを引数に渡すだけで簡単に動作検証を行えるようになります。
